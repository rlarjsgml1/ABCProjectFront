## 1~2. 화면별 평가 및 구체적 개선안

### 1) 홈 (U001Home)

**좋은 점**
- 히어로 슬라이드 구조(트랙 슬라이드 + 화살표 + 도트) 자체는 표준적인 캐러셀 UX를 잘 갖췄고, `hover`/`focus-within`에서만 화살표가 나타나는 처리(`home.css:150-159`)는 세련된 디테일입니다.
- 실제 도서 섹션(추천/신간/베스트)은 `book.coverImageUrl`이 있으면 이미지를, 없으면 색상 스와치를 쓰는 방어적 구조(`HomePage.tsx:298-302`)가 이미 마련되어 있습니다.
- 퀵메뉴 카드(`home-quick-menu`)는 아이콘+제목+설명 3단 구성과 hover 시 `translateY` 효과로 입체감을 주려는 시도가 보입니다.

**나쁜 점 (실제 원인 확인됨)**
- **히어로 "책 표지"가 실제 이미지가 아니라 CSS로 그린 사각형입니다.** `HomePage.tsx:218-222`의 `.home-hero-face`는 `banner.coverTitle` 텍스트와 `"ABC Book"`이라는 고정 문구만 표시하는 그라디언트 박스(`home.css:98-109`, 위 62%는 회색, 아래는 갈색 띠)입니다. 배너 데이터(`bannerItems`, `HomePage.tsx:28-47`)에 이미지 URL 필드 자체가 없습니다. 사용자가 "표지가 placeholder처럼 깨져 보인다"고 느낀 게 정확히 맞습니다 — 실제로 placeholder입니다.
- **카테고리 아이콘이 이름과 무관하게 배열 순서(`index % 8`)로만 매칭됩니다.** `categoryIcons = ['📖','💼','🏛','💻','🌱','✒️','🌿','⭐']`를 8개 고정 카테고리명 배열과 나란히 인덱스로 매핑(`HomePage.tsx:55-66, 262-267`)하는 구조라, API가 카테고리를 다른 순서/개수로 내려주는 순간 아이콘이 무조건 어긋납니다. 지금 눈에 보이는 "안 맞음"은 우연이 아니라 구조적으로 보장된 버그입니다. — **[업데이트] 홈 퀵메뉴가 이후 개편되면서 이 `categoryIcons`/`index % 8` 코드 자체가 현재 `HomePage.tsx`에는 존재하지 않습니다.** 퀵메뉴는 이제 `{ to, icon, label, description }` 형태의 고정 4개 항목(출석/이벤트/챌린지/카테고리)으로 바뀌어 있어, 원래 지적했던 "카테고리 이름-아이콘 어긋남" 버그는 현재 코드에서 재현되지 않습니다. 다만 이건 우연히 구조가 바뀌며 사라진 것이라 다시 카테고리별 동적 렌더링을 넣게 되면 동일한 함정(이름 기반 매칭이 아닌 인덱스 기반 매칭)에 다시 빠지지 않도록 주의가 필요합니다.
- 실제 표지가 없을 때 쓰는 폴백 색상(`tone: '#eda0a4'`, `'#e4eda4'`, `'#e34d9c'` 등, `HomePage.tsx:68-95`)이 브랜드 팔레트와 무관한 파스텔/마젠타 계열이라, 표지가 없는 책이 섞이면 화면이 알록달록 산만해집니다.
- 헤더/히어로/퀵메뉴/공지 티커가 각자 `#003d2d`를 로컬 변수로 재선언(`home.css:2`, `layout.css:3096` 등)하는 동안, 실제 버튼(`.button-primary`)은 `variables.css`의 파랑을 씁니다. 같은 화면 안에서 "포인트 컬러"가 두 개로 나뉘어 있으니 어디에도 힘이 실리지 않고 평면적으로 느껴집니다.

**개선 방법** — ✅ 1·3·4번 항목은 적용 완료(PR #112·#120), 2번은 대상 코드 소멸로 조치 불요.

- 정작 스타일은 `background: var(--color-surface-muted)`만 있고 테두리·그림자·포커스 강조가 없어서(확인된 CSS 범위 내), 헤더 배경과 거의 구분이 안 되는 "밋밋함"으로 이어집니다. 좋은 컴포넌트 구조인데 시각적 존재감이 없는 상태입니다.
- `SearchResultsPage`의 빈 상태(검색 전) 패널(`search.css`)은 글자 크기가 최대 3.35rem까지 커지는데 시각 요소는 텍스트뿐이라 "빈 벽"처럼 느껴집니다.

**개선 방법**
- 평상시: `border: 1px solid var(--color-border)` + `box-shadow: var(--shadow-sm)`로 존재감 부여. 포커스 시: 2px 브랜드 컬러 아웃라인 + `shadow-md`로 격상, 아이콘 컬러도 회색이 아닌 브랜드 컬러로.
- `search-request-panel`에 검색/책 아이콘 일러스트를 텍스트 위에 하나 추가해 "안내 문구만 있는 빈 화면" 느낌을 줄이세요.

### 5) 마이페이지 (U014MyPage)

**나쁜 점 — 사용자가 지적한 "빽빽함"이 코드로 확인됨**
- `MyPageOverview.tsx` 기준으로 한 화면에 **카운트 타일 6개(리뷰/포인트/쿠폰 + 대여/완독/즐겨찾기) + 동일 크기 위젯 4개(반납 알림/알림/출석/챌린지) + 최대 9개의 도서 미리보기 카드(진행률 바 포함)**가 전부 같은 배경색(`--mypage-widget-bg: var(--color-surface)`), 같은 radius, 같은 시각적 무게로 나열됩니다. ui-ux-pro의 대시보드 가이드라인도 "모든 지표를 동일 크기로 두는 것"을 안티패턴(Severity: High)으로 명시합니다 — 지금 구조가 정확히 그 사례입니다.
- 위젯 사이 간격(`--mypage-widget-gap: var(--space-3)` = 12px)이 좁아 카드 밀도가 더 높게 느껴집니다.

**개선 방법**
- **위계를 명시적으로 나누세요**: (1) 최우선 — "이어보기" 위젯(현재 읽던 책 표지+진행률)을 상단에 크게 단독 배치, (2) 2순위 — 카운트 6개는 지금처럼 그리드가 아니라 얇은 한 줄 스트립으로 축소, (3) 3순위 — 알림/출석/챌린지는 2열 유지하되, "반납 임박"처럼 실제로 주의가 필요한 항목에만 경고색 배경/테두리를 줘서 눈에 띄게 하고 나머지는 중립 배경으로.
- 위젯 내부 간격은 좁게 유지하되, **섹션과 섹션 사이 간격은 `space-6~8`로 넓혀** 그룹 구분을 시각적으로 만드세요.
- 7칸 출석 스탬프, 6칸 트로피 로우, 미니 바 리스트는 이미 옆의 독서 통계(reading-statistics)에서 Recharts를 쓰고 있으니, div 나열 대신 같은 차트 라이브러리로 통일하면 완성도가 올라갑니다.

### 6) 관리자 화면 (예: 회원 관리 A002Members, 전체 19개 화면 공통 패턴)

**좋은 점**
- 필터 행 + 테이블 + 상태 배지 + 모달이라는 표준 어드민 패턴 자체는 합리적입니다. 관리자 대시보드(`admin-dashboard-*`, `layout.css:357-510`)는 토큰(`--color-primary-soft`, `--color-success-soft` 등)을 성실히 참조하고 있어 앱 전체에서 **토큰 준수도가 가장 높은 화면**입니다.

**나쁜 점**
- `AdminMemberListPage.module.css` 하나에서만 상태 배지·테이블 헤더·액션 버튼 색이 전부 개별 hex로 하드코딩(`.statusJOINED #166534/#dcfce7`, 헤더 `#eaf2ff`, 액션 버튼 `#1d4ed8/#eff6ff/#dbeafe` 등, 271-364행)되어 있고, 이런 패턴이 19개 admin 모듈 CSS에 조금씩 다르게 반복됩니다. `variables.css`에 이미 `--color-success-soft`, `--color-danger-soft` 같은 대응 토큰이 있는데도 쓰이지 않고 있습니다.

**개선 방법**
- 공용 `.badge-success/-danger/-warning/-neutral` 클래스를 만들어 전 admin 모듈이 이걸 쓰게 하면, 발견된 하드코딩 hex 60여 개 중 상당수(배지/헤더 틴트 관련)가 한 번에 정리됩니다. 대시보드 화면의 토큰 사용 패턴을 사내 표준으로 문서화해 다른 admin 화면에 적용하는 걸 권장합니다.

### 7) 공통 — 회색 위 회색(대비) 문제

- `books.css`의 회색 플레이스홀더(`#969191`) 위 흰 텍스트(437행), `event.css`의 `rgba(15,23,42,0.58)` 회색 텍스트(133행), `home-notice-ticker` 배경(`#f6f7f8`) 위 `--color-muted`(#6b7280) 텍스트 등, **"muted 텍스트를 muted 배경 위에" 쓰는 조합이 여러 파일에서 반복**됩니다. 개별로는 WCAG를 겨우 통과할 수 있어도, 인접한 회색조 표면이 계속 이어지면서 화면 전체의 위계가 사라지는 효과를 냅니다.
- 개선: "표면(surface) 레벨"을 2단계(기본 surface / muted surface)로만 제한하고, 텍스트 대비가 필요한 곳은 muted 텍스트 대신 `--color-text`를 쓰거나, 반대로 표면 자체를 흰색으로 올리는 식으로 인접한 회색 evel이 3단 이상 쌓이지 않게 규칙을 두는 것을 권장합니다.

## 3. 색상 팔레트 추천 (3가지 옵션)

> **[결정 완료: 옵션 A 채택·적용]** PR #112에서 옵션 A(딥그린+앰버)가 적용됐습니다. `variables.css`는 primitive(`--green-900/700/100`, `--amber-500/100`) → semantic(`--color-primary`, `--color-accent` 등) 2계층으로 재구성됐고, 배경은 `#f8fbf9`(옅은 그린 틴트)가 됐습니다. 단, 어드민 콘텐츠 영역(`.admin-main`)은 후속 결정(PR #114)으로 그린 틴트 대신 흰색(`--color-surface`)을 사용합니다 — "배경·표면 흰색 고정" 공통 원칙의 어드민 강화판입니다. 옵션 B/C 원문은 추후 리브랜딩 시 참고용으로 유지합니다.

지금은 파랑(`variables.css`)과 그린(`#003d2d`, 각 페이지 로컬 변수)이 이중으로 섞여 있어 브랜드 색이 하나로 안 잡힌 상태입니다. "어느 색으로 통일할지"는 로고/브랜드 자산 유무에 따라 갈릴 수 있는 선택이라, 하나로 단정하기보다 3가지 방향을 제시합니다. 셋 다 아래 공통 원칙을 지키므로 어느 쪽을 골라도 적용 리스크는 동일합니다.

### 공통 원칙 (옵션과 무관하게 항상 적용)

- **배경·표면은 흰색 고정**: `--color-surface: #ffffff`는 유지하고, `--color-background`도 아주 옅은 오프화이트(흰색에 가까운 톤)로만 둡니다. 채도 있는 브랜드 컬러는 버튼·링크·배지·강조 요소 같은 "포인트"에만 쓰고, 카드/페이지 배경 같은 넓은 면적에는 쓰지 않습니다. 어두운 배경(다크 테마)은 다루지 않습니다.
- **토큰 이름은 그대로, 값만 교체**: `--color-primary`, `--color-primary-dark`, `--color-primary-soft`, `--color-accent`, `--color-accent-soft` 이름 체계를 유지한 채 값만 바꾸는 방식이라, 어느 옵션을 선택해도 적용 절차(2번 섹션들에서 언급한 "포인트 컬러 통일" 작업)는 동일합니다.
- **신호색(성공/실패/경고)은 옵션과 무관하게 고정**: 사용자가 "초록=성공, 빨강=오류"로 이미 학습되어 있으므로, 표준 그린/레드/앰버 계열(`--color-success`, `--color-danger`, `--color-warning`)은 세 옵션 모두 손대지 않습니다.

### 옵션 A — 딥그린 + 앰버 (기존 관성 유지, 변화 최소)

이미 헤더/홈/검색/도서목록/이벤트 페이지가 전부 `#003d2d`를 자체 변수로 쓰고 있다는 것 자체가, 실제 브랜드 색은 이미 그린으로 합의되어 있다는 신호입니다. 그린을 버리기보다 `--color-primary`를 그린으로 공식화하는 쪽이 지금까지의 시각적 관성과 가장 잘 맞습니다.

```css
:root {
  --green-900: #003d2d;   /* 기존 --home-green, --books-green, --abc-header-brand 통합 */
  --green-700: #00684d;   /* 기존 --home-green-bright, hover/active */
  --green-100: #e3f5ee;   /* soft 배경 */
  --amber-500: #e3a008;   /* 강조 accent — 배지, 별점, 프로모션 */
  --amber-100: #fdf0d5;

  --color-primary: var(--green-900);
  --color-primary-dark: var(--green-700);
  --color-primary-soft: var(--green-100);
  --color-accent: var(--amber-500);
  --color-accent-soft: var(--amber-100);
  --color-background: #f8fbf9;
  --color-surface: #ffffff;
}
```

**적합한 경우**: 화면 인상을 거의 바꾸지 않으면서 "이미 쓰고 있던 색을 공식 토큰으로 승격"만 하고 싶을 때. 리스크가 가장 낮은 1순위 후보.

### 옵션 B — 웜 테라코타 + 딥틸 (독서 서비스다운 편집적 무드)

파랑 계열 SaaS 템플릿 느낌에서 가장 멀리 벗어나는 방향입니다. 서점·도서관 특유의 따뜻하고 편집적인 톤으로, "데모 느낌"을 벗어나는 효과가 가장 큽니다.

```css
:root {
  --color-primary: #b5502f;
  --color-primary-dark: #8c3c22;
  --color-primary-soft: #fbeae2;
  --color-accent: #2f6b5e;       /* 딥틸 보조 포인트 */
  --color-accent-soft: #e3eeec;
  --color-background: #fdfbf8;
  --color-surface: #ffffff;
}
```

**적합한 경우**: "튜토리얼/데모 템플릿 인상 탈피"가 최우선 목표일 때. 다만 지금 화면 곳곳에 이미 쓰인 그린 자산(로고, 아이콘 배경 등)이 있다면 함께 손볼 범위가 세 옵션 중 가장 큽니다.

### 옵션 C — 네이비/인디고 (전문적·무난, 관리자 화면과 궁합 good)

가장 "실패 확률이 낮은" 선택지입니다. 관공서·금융 느낌까지는 아니지만 신뢰감 있는 톤이라, 유저 화면과 관리자 대시보드를 같은 색으로 묶기에 부담이 없습니다.

```css
:root {
  --color-primary: #3949ab;
  --color-primary-dark: #283593;
  --color-primary-soft: #e8eaf6;
  --color-accent: #ffb300;       /* 밑줄/뱃지 포인트 */
  --color-accent-soft: #fff3d6;
  --color-background: #f7f8fc;
  --color-surface: #ffffff;
}
```

**적합한 경우**: 무난하고 안전하게 가고 싶을 때, 또는 admin 대시보드까지 톤을 통일하고 싶을 때. 다만 기존 그린 자산과는 결이 가장 멀어, 로고 등 기존 브랜드 요소와 같이 쓸 경우 충돌 여부를 별도 확인해야 합니다.

### 선택 가이드

| 우선순위 | 추천 옵션 |
|---|---|
| 변화 최소화, 기존 그린 자산 보존 | **옵션 A** |
| "데모 느낌" 탈피가 최우선 | **옵션 B** |
| 무난·안전, admin까지 통일 | **옵션 C** |

- 파랑(`#2563eb`)은 세 옵션 모두에서 폐기 대상입니다. 굳이 남기려면 "정보/링크" 전용 보조색으로 admin 정보성 배지에만 제한적으로 쓰는 정도로 격하하세요.
- `.button-primary`, 헤더 브랜드, 히어로 배지, mypage 강조 위젯 등 지금 파랑/그린이 섞여 있던 모든 지점을 선택한 옵션의 `--color-primary` 하나로 통일하면 "포인트 컬러 없음" 문제가 상당 부분 해결됩니다.
- 도서 목록/이벤트의 임의 gray(`#969191`, `#4d4d4d`, `#e5e7eb`, `#777` 등 파편화된 회색들)는 옵션과 무관하게 slate 계열 5~6단 스케일로 축소해 재사용하세요.

## 4. 3D 요소 / 모션 제안 (framer-motion)

전자책 대여 서비스는 콘텐츠/텍스트 밀도가 높은 서비스이므로, **풀 3D(Three.js 등)는 과할 위험이 크고 성능 비용 대비 효과가 낮습니다.** 대신 framer-motion 기반의 **가벼운 모션 + 선택적 CSS 3D 틸트** 정도가 톤에 맞습니다.

> **적용 현황**: 아래 8개 항목 중 7개가 이미 반영되어 있습니다. 남은 건 뷰어 페이지 넘김 하나뿐입니다. 전반적으로 doc이 제안한 강도(2~4도 틸트, `whileTap` 스프링, `whileInView` stagger)를 그대로 따랐고 과하게 나가지 않은 점이 좋습니다.

| 위치 | 제안 | 효과 | 상태 |
|---|---|---|---|
| 홈 히어로 캐러셀 | 현재 CSS `transform` 트랜지션을 framer-motion `AnimatePresence`로 교체, 슬라이드 전환 시 살짝 페이드+슬라이드 | 전환이 더 부드럽고 통제 가능 | ✅ `HomePage.tsx:262-285` |
| 홈 섹션(카테고리/베스트/신간) | `whileInView`로 스크롤 진입 시 카드가 살짝 아래→위로 stagger 페이드인 | "정적인 페이지" 인상을 가장 크게 개선 | ✅ `HomePage.tsx:351-371` |
| 도서 커버(목록/베스트 그리드) | hover 시 CSS 3D `perspective` + `rotateX/Y` 2~4도 틸트(가벼운 카드 기울임) | 저비용으로 "입체감" 부여, 3D 엔진 불필요 | ✅ `HomePage.css:428-443`, `BooksPage.css:443-461` (실측 `rotateX(4deg) rotateY(-4deg)`) |
| 즐겨찾기 하트 | `whileTap`으로 스프링 바운스 스케일 | 클릭 피드백 강화 | ✅ `BookDetailPage.tsx:478-487` (`stiffness: 400, damping: 15`) — active 상태 대비 부족 문제도 `--color-primary` 적용으로 함께 해결됨 |
| 마이페이지 카운트 타일 | 진입 시 숫자 카운트업 애니메이션 | 데이터가 살아있다는 느낌 | ✅ `CountUpValue.tsx` 컴포넌트화 |
| 검색창 포커스 | `focus`/`hover` 시 스케일 1→1.01 + 그림자 확장 | 앞서 지적한 "밋밋함" 보완 | ✅ `layout.css:3348-3376` |
| 뷰어 페이지 넘김 | 페이지 전환 시 짧은 슬라이드/페이드 | 실제 책 넘기는 느낌 강화 | ✅ 적용 완료 — epub.js iframe 구조 제약으로 컨테이너 150ms 페이드 방식(`turnPage` 헬퍼), PRM 대응 |
| 관리자 대시보드 KPI | 카드 진입 애니메이션 + 스켈레톤 로딩 | 데이터 로딩 중 공백 최소화 | ✅ 적용 완료 — 진입 애니메이션 + 스켈레톤(8카드 shimmer, `admin-dashboard-kpi-skeleton`, PRM 대응)까지 확인됨 |

**3D는 "히어로 책 표지" 한 곳에만 제한적으로** — 실제 이미지가 준비되면 CSS 3D perspective로 살짝 기울여 놓고 마우스를 따라 미세하게 반응하는 정도(parallax tilt)면 충분합니다. Three.js 같은 풀 3D는 로딩/번들 크기 대비 이 서비스 성격(텍스트 중심, 반복 방문형)에는 권장하지 않습니다. (도서 커버 틸트는 이미 이 원칙대로 CSS만으로 구현되어 있어 방향이 맞습니다.)

**완료 노트**: 이 표의 8개 항목이 전부 적용되어 4번 섹션은 종결됐습니다. 뷰어 넘김은 "아주 짧게(150ms 이하)" 권고대로 150ms 페이드로 구현했고, epub.js가 iframe 내부를 직접 렌더링하는 구조라 framer-motion 대신 컨테이너 페이드 방식을 택했습니다. 관리자 KPI 스켈레톤은 재확인 결과 이미 shimmer 애니메이션까지 구현되어 있었습니다(이전 "미확인" 표기는 오기).

## 5. 반응형(Responsive) vs 적응형(Adaptive)

> **적용 현황: ✅ 섹션 종결.** "반응형 유지" 방향 확정에 이어 브레이크포인트 표준화도 적용 완료됐습니다 — 22개 CSS 파일의 15가지 값(560~1440px)을 **768/1024/1280/1536** 4단으로 일괄 치환하고, `variables.css` 상단에 팀 규칙 주석을 고정했습니다. 이로 인해 중간 폭(태블릿 구간)에서 레이아웃이 접히는 시점이 일부 달라졌습니다(예: 640px에서 접히던 화면이 768px부터 접힘) — 표준화의 의도된 결과입니다.

**반응형을 유지/강화하는 것을 권장합니다.**

이유:
- 이미 `clamp()`, `minmax()`, `repeat(auto-fill/auto-fit, ...)` 같은 유동 그리드 기법을 홈/도서목록/도서상세 전반에서 쓰고 있어(`home.css`, `BookDetailPage.module.css` 등) 반응형과 궁합이 맞는 기반이 이미 있습니다.
- 페이지 수가 47개(유저 29 + 관리자 19)에 달해, 적응형처럼 디바이스 구간별 별도 레이아웃을 다 만들면 유지보수 비용이 배가됩니다.
- 적응형이 유리한 경우(디바이스별로 완전히 다른 인터랙션이 필요한 경우, 예: 네이티브 앱 전용 UI)에 해당하지 않습니다 — 이 프로젝트는 단일 웹으로 전 디바이스를 커버합니다.

**브레이크포인트 정리 제안**

현재 코드 전반에 걸쳐 `640, 720, 760, 768, 900, 960, 1180, 1200, 1440`처럼 **파일마다 제각각인 브레이크포인트**가 흩어져 있습니다(예: `admin-shell`은 900px에서 접히는데 `admin-dashboard-kpi-grid`는 1200px에서 접혀서, 같은 admin 레이아웃 안에서도 "좁다"의 기준이 서로 다릅니다). 아래처럼 표준 4~5단으로 합치는 것을 제안합니다.

```css
/* 참고용 상수 — CSS 커스텀 프로퍼티는 미디어쿼리 조건에 직접 못 쓰므로,
   주석/문서로 팀 규칙을 통일하고 실제 값은 그대로 숫자로 사용 */
/* --bp-sm: 480px  (큰 폰) */
/* --bp-md: 768px  (태블릿 세로) */
/* --bp-lg: 1024px (태블릿 가로 / 소형 노트북) */
/* --bp-xl: 1280px (데스크톱) */
/* --bp-2xl: 1536px (와이드 데스크톱, admin 대시보드 등 정보 밀도 높은 화면) */
```

- 유저 화면: `768 / 1024 / 1280` 3단으로 충분.
- 관리자 화면(테이블 위주라 더 넓은 폭이 필요): `1024 / 1280 / 1536` 기준으로 통일 — 지금처럼 900과 1200이 따로 노는 걸 정리.

**적용 결과**: 위 제안과 달리 점진 방식이 아닌 일괄 치환으로 진행했습니다(미디어쿼리 형식이 `@media (max-width: NNNpx)`로 균일하고 min-width 쿼리가 없어 기계적 치환이 안전했음). 매핑: 560~760→768, 860~980→1024, 1100~1200→1280, 1440→1536.

## 6. 폰트 페어링 추천

> **적용 현황: ✅ 적용 완료.** 아래 제안대로 Pretendard(본문) + Noto Serif KR(헤딩) 페어링이 반영됐고(`index.html:8-11`에서 구글폰트/jsdelivr CDN 로드, `variables.css:28-29`에 `--font-heading`/`--font-body` 토큰 정의), 본문 크기도 15.2px→16px(1rem)로 올라갔습니다(`variables.css:34`). 아래 원문 설명과 표는 "왜 이렇게 제안했는지" 기록으로 남기고, 실제 반영된 값은 표 아래 갱신 표로 대체합니다.

현재 `base.css`는 `font-family: Inter, system-ui, ...` 단일 스택입니다. 두 가지 짚을 점이 있습니다.

1. **Inter는 한글 글리프가 없습니다.** 이 서비스 UI 텍스트 대부분이 한글이므로, 실제로는 Inter가 거의 적용되지 않고 OS 기본 한글 폰트(Windows: 맑은 고딕, Mac: Apple SD 산돌고딕)로 폴백되고 있을 가능성이 높습니다. 즉 지금 폰트 선언은 숫자/영문에만 실질적으로 작동하고, 한글 타이포는 사실상 관리되지 않는 상태입니다.
2. 지금은 UI 폰트 하나로 마케팅성 히어로 헤드라인부터 표/배지까지 다 처리하고 있어, "독서 서비스다운" 편집적 톤이 없습니다.

**제안 (ui-ux-pro 폰트 페어링 데이터베이스 참고, 한글 대응으로 조정)**

- **본문/UI**: `Pretendard`(한글 최적화, Inter와 유사한 기하학적 그로테스크 계열이라 지금 톤 유지하면서 한글도 제대로 렌더링됨) → `Inter`를 라틴/숫자 폴백으로 유지, `system-ui`를 최종 폴백으로.
- **헤딩/히어로/도서 제목**: 세리프 계열을 하나 추가해 "읽는 서비스"다운 편집 감성 부여 — `Noto Serif KR`(무료, 구글 폰트, 한글 세리프 중 완성도 높음) 또는 이미 라이선스가 있다면 리디바탕류. ui-ux-pro의 "Editorial Classic"(Cormorant Garamond + Libre Baskerville), "News Editorial"(Newsreader + Roboto) 페어링이 참고 방향과 일치합니다 — 헤딩에 세리프, 본문/UI에 산세리프.

```css
:root {
  --font-heading: 'Noto Serif KR', 'Pretendard', serif;
  --font-body: 'Pretendard', Inter, system-ui, -apple-system, 'Segoe UI', sans-serif;
}
```

- 적용 범위: 홈 히어로 `h1`, 도서상세 제목, 이벤트/공지 배너 헤드라인에는 `--font-heading`, 나머지 UI/폼/테이블/관리자 화면은 전부 `--font-body` 유지 — 관리자 화면까지 세리프를 넣으면 오히려 가독성이 떨어지므로 admin은 현재 산세리프 톤을 그대로 가져가는 걸 권장합니다.

### 폰트 사이즈 스케일 — ✅ 적용 완료, 실제 반영된 값 (`variables.css:30-37` 기준)

원래 제안했던 표는 아래로 대체합니다(제안값과 실제 적용값이 소폭 다릅니다 — base를 1rem으로 올린 건 동일하지만, 각 단계 간격을 제안보다 조금 더 벌려 위계를 더 뚜렷하게 잡았습니다).

| 토큰 | 값 | 비고 |
|---|---|---|
| `--font-size-xs` | 0.8rem (12.8px) | 캡션, 뱃지, 타임스탬프 |
| `--font-size-sm` | 0.9rem (14.4px) | 보조 텍스트, 테이블 셀 |
| `--font-size-base` | **1rem (16px)** | 본문 — 제안대로 16px로 상향 완료 |
| `--font-size-md` | 1.1rem (17.6px) | 강조 본문, 폼 라벨 |
| `--font-size-lg` | 1.2rem (19.2px) | 카드/위젯 제목 |
| `--font-size-xl` | 1.35rem (21.6px) | 섹션 제목 |
| `--font-size-2xl` | 1.7rem (27.2px) | 페이지 제목 |
| `--font-size-3xl` | 2.1rem (33.6px) | 히어로 헤드라인 |

**의견**: 이 정도면 충분합니다. 원래 우려였던 "본문 15.2px는 읽기 서비스 치고 작다"는 문제가 해결됐고, 단계 간 비율도 이전보다 명확해져서 제목/본문 위계가 더 잘 드러날 것으로 보입니다. `font-display: swap`은 구글폰트 링크(`index.html:10`)에 이미 `&display=swap`이 붙어 있어 FOIT 문제도 함께 해결된 상태입니다. 다만 jsdelivr로 받아오는 Pretendard 링크(`index.html:11`)는 `display=swap` 파라미터가 없는 서브셋 CSS라 — 실제로 FOIT이 발생하는지는 느린 네트워크(스로틀링)에서 한 번 눈으로 확인해보는 걸 권합니다.

**반응형 폰트**: 히어로 헤드라인(`--font-size-3xl`)처럼 큰 사이즈는 고정 값 대신 `clamp(1.6rem, 4vw + 1rem, 2rem)`처럼 `clamp()`를 쓰면 좁은 화면에서 줄바꿈이 어색해지는 걸 방지할 수 있습니다. 반대로 본문/UI 텍스트(`base`~`lg`)는 화면 크기와 무관하게 고정 rem을 유지하는 편이 가독성에 더 유리합니다 — 뷰포트에 따라 본문 글자가 계속 작아지면 오히려 읽기 어려워집니다.

## 7. 공통 CSS 구조 제안 — "화면별 모듈" 유지 + 디자인 토큰 레이어 보강

**결론: 지금의 CSS Module(화면별) + 전역 유틸(`base.css`/`layout.css`) 하이브리드 구조 자체는 바꿀 필요 없습니다.** 47개 화면을 새 아키텍처로 전면 재작성하는 건 리스크 대비 이득이 작습니다. 문제는 구조가 아니라 **토큰 레이어가 얇고, 그 얇은 토큰마저 지켜지지 않는다**는 것입니다. 실제로 `variables.css`/`*.css`/`*.tsx` 전체에서 하드코딩 hex가 60개 이상 발견됐고, 그중 다수가 admin 모듈 CSS의 배지/테이블 색입니다.

**제안하는 단계적 보강**

1. ✅ **토큰을 2계층으로 분리** (PR #112에서 적용 완료): primitive(원색 팔레트) → semantic(용도별 이름). 지금 `--color-primary` 하나에 여러 로컬 변수(`--home-green`, `--books-green`, `--abc-header-brand`)가 각자 같은 값을 재선언하는 걸 없애고 전역 semantic 토큰 하나만 참조하게 합니다(3번 팔레트 섹션의 코드 예시 참고). → 실제 적용에서는 로컬 변수명을 유지한 채 값을 전역 토큰 별칭(`--home-green: var(--color-primary)`)으로 바꾸는 방식을 택해, 사용처 코드 변경 없이 단일 출처를 달성했습니다.
2. **공용 프리미티브 클래스 소량 추가**: `.badge-success/-danger/-warning/-neutral`, `.cover-placeholder`, `.card-surface` 등 3~5개 정도만 `base.css`나 `layout.css`에 추가하고, admin 19개 모듈이 새 배지를 만들 때마다 색을 새로 정의하지 않고 이 클래스를 쓰게 유도합니다. 전면 교체가 아니라 신규/수정 시점에 점진적으로 옮기는 방식이면 충분합니다.
3. **브레이크포인트 값을 문서화된 상수로 통일**(5번 섹션 참고) — CSS 변수를 미디어쿼리 조건에 직접 못 쓰는 한계가 있으므로, PostCSS 플러그인(`postcss-preset-env`의 custom media, 또는 SCSS 도입)을 검토하면 실제 변수로 강제할 수 있습니다. 당장 어렵다면 최소한 팀 컨벤션 문서로 통일.
4. **린트로 회귀 방지**: stylelint에 `color`/`background` 계열 속성에 raw hex 리터럴을 금지하는 규칙(`declaration-property-value-disallowed-list`)을 추가해, 토큰 정리 후 새 하드코딩 색이 다시 섞이는 걸 CI에서 막는 것을 권장합니다.

이 정도 보강이면 구조를 갈아엎지 않고도 "포인트 컬러 통일 / 회색 대비 정리 / admin 배지 일관성" 문제 대부분이 해결됩니다.

## 8. 좋은 웹사이트를 위해 함께 챙기면 좋은 것들

색상/타이포/모션 외에, 실제 서비스 완성도에 영향을 주는 항목들입니다. 화면틀을 바꾸지 않고도 적용 가능한 것 위주로 정리했습니다.

- **명도 대비(WCAG AA, 4.5:1 이상)**: 특히 옵션 B/C처럼 채도가 강한 컬러를 텍스트에 쓸 경우, 흰 배경 위에서 실제 대비를 체크하세요. `--color-muted` 텍스트를 muted 배경 위에 겹쳐 쓰는 조합(7번 섹션에서 지적한 "회색 위 회색" 문제)이 이 기준을 가장 자주 어깁니다.
- **키보드 포커스 표시**: 버튼·링크·입력창에 `:focus-visible` 아웃라인이 브랜드 컬러로 명확히 보이는지 확인하세요. 마우스 사용자에게는 안 보여도 키보드 사용자에게는 필수입니다. — ✅ PR #116에서 전역 규칙으로 적용 완료.
- **색만으로 의미를 전달하지 않기**: 상태 배지(성공/실패/경고)에 색뿐 아니라 아이콘이나 텍스트 라벨을 함께 표시하세요 — 색맹 사용자 배려 차원입니다.
- **로딩/빈 상태**: 카운트 타일, 도서 목록 등 API 응답 전에 스켈레톤 UI를 넣으면 "빈 화면 깜빡임"이 줄어듭니다. 1번 섹션에서 지적한 표지 placeholder 개선과 같은 맥락입니다.
- **터치 타겟 크기**: 모바일에서 버튼/아이콘 클릭 영역을 최소 44×44px 확보하세요. 유저 화면의 하트/북마크 아이콘처럼 작은 터치 영역은 별도 확인이 필요합니다.
- **이미지 lazy loading + 명시적 크기**: 도서 표지 이미지에 `loading="lazy"`와 `width`/`height`(또는 `aspect-ratio`)를 지정해 레이아웃 밀림(CLS)을 방지하세요.
- **웹폰트 로딩**: Pretendard/Noto Serif KR 적용 시 `font-display: swap`을 설정해, 폰트 로딩 중 텍스트가 안 보이는 현상(FOIT)을 방지하세요. — ✅ 구글폰트(Noto Serif KR) 쪽은 `index.html:10`에 `&display=swap`이 반영됨. jsdelivr Pretendard 서브셋 CSS(`index.html:11`)는 파라미터로 확인이 안 되니 느린 네트워크에서 한 번 체크 권장.
- **모션 접근성**: `prefers-reduced-motion: reduce`를 감지해 framer-motion 애니메이션을 최소화하는 분기를 추가하면, 모션에 민감한 사용자를 배려할 수 있습니다. — ✅ 적용 완료. 4번 섹션에서 추가된 모든 모션 컴포넌트가 `useReducedMotion`/media query로 이 원칙을 지키고 있습니다.
- **다크 모드는 이번 제안 범위 밖**입니다. "너무 어둡지 않은 톤"을 원하신다는 방향과도 맞게, 3번 섹션의 팔레트 3안은 전부 라이트 테마 전용으로 설계했습니다. 추후 필요하면 별도로 검토하는 것을 권장합니다.
